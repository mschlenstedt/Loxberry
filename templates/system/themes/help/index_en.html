Overview / concept

The PDF uses the preview as visual reference: forms, selects, multiselect, sliders, buttons, cards and tables are shown as real Core components. This help page follows that order and adds recommended developer markup.

Core remains plugin-neutral. Plugin-specific visuals belong in plugin or theme CSS. New generic helpers should usually go to utilities.css; component defaults belong to components.css.

  • Core comments and Core classes remain plugin-neutral.
  • Theme-specific decisions belong to the corresponding theme CSS.
  • Plugin CSS fixes plugin markup, not the global framework.

Files and responsibilities

The separation enables future themes without changing every plugin again.

Components / components.css

components.css contains the actual default rendering for LoxBerry Design System components. Plugin developers should use these classes directly in markup instead of rebuilding the visual style of individual controls in plugin CSS. Colors, radii, shadows and contrast still come from design-tokens.css and from the active theme.

The separation is important: components.css defines stable building blocks such as card, form row, input, select, button, toggle, slider and table. utilities.css only adds optional behavior such as layers, overflow, width, spacing or table frames. Theme files change design values, not the basic structure of a component.

Rule of thumb for developers

  • Use a Core component first when the UI element already exists.
  • Add only missing plugin-specific layout in the plugin CSS.
  • Do not hard-code colors when a --lb-* token exists.
  • Do not force optional helpers into components.css; use utilities.css for that.

Plugin page structure

For new plugin pages, a minimal and stable base pattern is better than many inline styles. The page loads Core CSS, sets the theme class on body and uses Core classes for layout, cards, forms and tables.

On real plugin CGIs, header and footer are still generated by LoxBerry. This example deliberately shows only the content-area markup pattern.

<!-- Minimal content area of a plugin page -->
<div class="lb-card">
  <div class="lb-section-title">Settings</div>

  <div class="lb-form-row">
    <label class="lb-form-label" for="plugin_option">Option:</label>
    <div class="lb-form-field">
      <input class="lb-input" id="plugin_option" name="plugin_option" data-role="none">
      <div class="lb-form-help">Short help text for users.</div>
    </div>
  </div>

  <div class="lb-actions">
    <button class="lb-btn lb-btn-primary" type="submit" data-role="none">Save</button>
  </div>
</div>

Tokens / themes

Developers should use tokens wherever possible, not fixed colors. This keeps Soft Rounded, Classic Mac, Liquid Glass and later user themes working without plugin-specific special cases.

Common tokens

--lb-primary--lb-text--lb-text-secondary--lb-card-bg--lb-border-color--lb-radius-md--lb-input-bg--lb-input-text--lb-btn-bg--lb-btn-primary-bg--lb-table-header-bg--lb-table-bg
/* Good: token based and themeable */
.my-plugin-box {
  background: var(--lb-card-bg);
  color: var(--lb-text);
  border: 1px solid var(--lb-border-color);
  border-radius: var(--lb-radius-md);
}

/* Only when really plugin-specific */
body.theme-liquid-glass .my-plugin-box {
  background: var(--lb-glass-card-bg, var(--lb-card-bg));
}

Developer checklist
  • Use a Core class before writing plugin CSS.
  • Use data-role="none" for native controls on jQM pages.
  • Solve colors, radii, shadows and backgrounds via tokens.
  • Make dropdowns/tooltips visible in tables only through opt-in utilities.
  • Check disabled fields: native disabled is not submitted.
  • Do not put plugin-specific rules into Core files.
  • Put theme-specific rules into theme CSS.
/* Troubleshooting order for rendering issues */
1. Is the markup a Core component?
2. Is jQuery Mobile still interfering?
3. Is data-role="none" missing?
4. Is the issue plugin-specific, theme-specific or generally Core-related?
5. Does the fix belong to components.css, utilities.css, theme CSS or plugin CSS?
