Overview / Konzept

Das PDF zeigt die Preview als visuelle Referenz: Formularfelder, Selects, Multiselect, Slider, Buttons, Cards und Tabellen erscheinen in echten Core-Komponenten. Die Hilfe folgt derselben Reihenfolge, erklärt aber zusätzlich, welches Markup Plugin-Entwickler verwenden sollen.

Core bleibt plugin-neutral. Plugin-spezifische Optik gehört in Plugin- oder Theme-CSS. Neue allgemeine Helfer gehören bevorzugt in utilities.css; Komponenten-Defaults gehören in components.css.

  • Core-Kommentare und Core-Klassen bleiben plugin-neutral.
  • Theme-spezifische Entscheidungen gehören in die jeweilige Theme-CSS.
  • Plugin-CSS korrigiert Plugin-Markup, nicht das globale Framework.

Dateien und Verantwortung

Die Trennung macht spätere Themes möglich, ohne dass jedes Plugin wieder angepasst werden muss.

Komponenten / components.css

components.css ist die Datei für das eigentliche Default-Rendering der LoxBerry-Design-System-Komponenten. Plugin-Entwickler sollen diese Klassen direkt im Markup verwenden, statt die Optik einzelner Controls im Plugin nachzubauen. Farben, Radien, Schatten und Kontraste kommen dabei weiterhin aus design-tokens.css bzw. aus dem aktiven Theme.

Wichtig ist die Trennung: components.css beschreibt stabile Bausteine wie Card, Form Row, Input, Select, Button, Toggle, Slider oder Tabelle. utilities.css ergänzt nur optionales Verhalten wie Layer, Overflow, Breite, Spacing oder Tabellenrahmen. Theme-Dateien ändern Designwerte, aber nicht die Grundstruktur einer Komponente.

Faustregel für Entwickler

  • Für ein bekanntes UI-Element zuerst eine Core-Komponente verwenden.
  • Nur fehlendes Plugin-spezifisches Layout in der Plugin-CSS ergänzen.
  • Keine Farben hart setzen, wenn ein --lb-*-Token existiert.
  • Keine optionalen Helfer in components.css erzwingen; dafür utilities.css nutzen.
<div class="lb-card">
  <div class="lb-section-title">Plugin-Einstellung</div>

  <div class="lb-form-row">
    <label class="lb-form-label" for="demo">Name:</label>
    <div class="lb-form-field">
      <input class="lb-input" id="demo" name="demo" data-role="none">
      <div class="lb-form-help">Kurzer Hinweis zum Feld.</div>
    </div>
  </div>

  <div class="lb-actions">
    <button class="lb-btn lb-btn-primary" type="submit" data-role="none">Speichern</button>
  </div>
</div>

Plugin-Seitenaufbau

Für neue Plugin-Seiten ist ein minimales, stabiles Grundmuster besser als viele Inline-Styles. Die Seite lädt Core-CSS, setzt die Theme-Klasse am Body und verwendet Core-Klassen für Layout, Cards, Formulare und Tabellen.

In echten Plugin-CGIs wird Header/Footer weiterhin von LoxBerry erzeugt. Dieses Beispiel zeigt bewusst nur das Markup-Prinzip für den Inhaltsbereich.

<!-- Minimaler Inhaltsbereich einer Plugin-Seite -->
<div class="lb-card">
  <div class="lb-section-title">Einstellungen</div>

  <div class="lb-form-row">
    <label class="lb-form-label" for="plugin_option">Option:</label>
    <div class="lb-form-field">
      <input class="lb-input" id="plugin_option" name="plugin_option" data-role="none">
      <div class="lb-form-help">Kurzer Hilfetext für Benutzer.</div>
    </div>
  </div>

  <div class="lb-actions">
    <button class="lb-btn lb-btn-primary" type="submit" data-role="none">Speichern</button>
  </div>
</div>

Selects / Multiselect

Selects sind ein Hauptthema der Preview. Native Selects verwenden lb-select. Der Core-Multiselect ist details-basiert und funktioniert ohne jQM-Selectmenu.

Tokens / Themes

Entwickler sollen nach Möglichkeit Tokens verwenden, nicht konkrete Farben. Dadurch funktionieren Soft Rounded, Classic Mac, Liquid Glass und spätere User-Themes ohne Plugin-Sonderfälle.

Häufige Tokens

--lb-primary--lb-text--lb-text-secondary--lb-card-bg--lb-border-color--lb-radius-md--lb-input-bg--lb-input-text--lb-btn-bg--lb-btn-primary-bg--lb-table-header-bg--lb-table-bg
/* Gut: tokenbasiert und themefähig */
.my-plugin-box {
  background: var(--lb-card-bg);
  color: var(--lb-text);
  border: 1px solid var(--lb-border-color);
  border-radius: var(--lb-radius-md);
}

/* Nur wenn wirklich plugin-spezifisch nötig */
body.theme-liquid-glass .my-plugin-box {
  background: var(--lb-glass-card-bg, var(--lb-card-bg));
}

Entwickler-Checkliste
  • Core-Klasse verwenden, bevor Plugin-CSS geschrieben wird.
  • Bei nativen Controls auf jQM-Seiten data-role="none" setzen.
  • Farben, Radius, Schatten und Hintergründe über Tokens lösen.
  • Dropdowns/Tooltips in Tabellen nur über opt-in Utilities sichtbar machen.
  • Disabled-Felder prüfen: native disabled wird nicht submitted.
  • Plugin-spezifische Regeln nicht in Core-Dateien schreiben.
  • Theme-spezifische Regeln in Theme-CSS ablegen.
/* Prüfreihenfolge bei Darstellungsproblemen */
1. Ist das Markup eine Core-Komponente?
2. Greift jQuery Mobile noch dazwischen?
3. Fehlt data-role="none"?
4. Ist das Problem plugin-spezifisch, theme-spezifisch oder Core-allgemein?
5. Gehört die Korrektur in components.css, utilities.css, Theme-CSS oder Plugin-CSS?
